# Feature Specification: 真實台指期資料源——TAIFEX 主源 + FinMind 交叉驗證

**Feature Branch**: `010-taifex-real-data`

**Created**: 2026-07-17

**Status**: Implemented

**Input**: User description: "讓期貨脫離 mock：接入真實台指期（TX）歷史與每日資料，run_ingestion 存真 fut_* 表，回測與監控用真資料；003 推播的『待真源生效』條款兌現。TAIFEX 官方為主源（權威、免費、1998 起）、FinMind 為交叉驗證哨兵；連續月拼接採成交量最大月轉倉 + back-adjust。"

> **資料源事實（2026-07-17 經瀏覽器/端點實測驗證）**：TAIFEX 歷史日線可回溯 1998-07-21
>（TX 上市日）、按「契約月份 × 交易時段」逐列、含結算價與未平倉；批次下載單次限一個月；
> 官方 OpenAPI 僅提供最近一交易日（適合增量、不能回填）。Yahoo/yfinance **無**台指期貨
> 代碼（`TXF=F` 不存在）。FinMind 為同源清洗鏡像（1998 起、token 600 req/hr）。

## Clarifications

### Session 2026-07-17

- Q: 歷史回填預設抓多深？ → A: 全歷史（1998-07-21 TX 上市日起）——一次到位（~340 請求 ≈ 12 分鐘、幾 MB），涵蓋完整多空循環；`backfill_start` config 可調。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 真實台指期資料端到端（Priority: P1）🎯 MVP

研究者執行資料匯入後，系統自 TAIFEX 官方取得台指期（TX）**真實**日線：歷史回填
（可指定起始年，逐月分段、保守節流）與每日增量（官方公開端點）皆自動化；
原始「按契約月份」資料與拼接後的**連續序列**分層入庫；既有回測與監控**零程式修改**
即消費真資料——監控訊息不再帶 mock 標示（003「待真源生效」條款兌現）。

**Why this priority**: 這是本 spec 的存在理由——沒有真資料，多空回測與推播都停在
mock 驗證層；002/007 的 FVG/MSS 懸案（SC-003 資料相依）也需要真實日內外資料才能推進。

**Independent Test**: 對指定小區間執行真實回填（需網路、測試預設跳過）→ 原始表與
連續表皆有資料、筆數與區間相符、回測跑通；解析層以離線真實樣本 fixture 驗證（免網路）。

**Acceptance Scenarios**:

1. **Given** 指定歷史區間（如 2023-01 ~ 2023-03），**When** 執行回填，**Then** 逐月取得 TAIFEX 資料（每請求間保守節流）、過濾為一般交易時段、原始按契約月份列存入原始表。
2. **Given** 原始資料已入庫，**When** 產生連續序列，**Then** 連續日線存入既有期貨表、通過資料品質契約（欄位完整、時間遞增、無缺值）、回測引擎直接可用。
3. **Given** 每日收盤後執行增量更新，**When** 官方當日資料可得，**Then** 當日列附加入庫、連續序列隨之更新。
4. **Given** TXF 資料源已切換為真源，**When** 監控執行，**Then** 訊息**無** mock 標示（003 條款兌現）、其餘行為不變。
5. **Given** TAIFEX 回應格式異常或請求失敗，**When** 匯入執行，**Then** 明確報錯（有限次重試後 fail-fast），不得靜默入庫壞資料。

### User Story 2 - 連續月拼接正確性（Priority: P2）

系統以**成交量最大月**規則選擇近月：當次月成交量超過現行近月時轉倉（單向、不回切、
每日判定、純資料內生 → 確定性可重現）；轉倉日以新舊契約收盤差**回溯平移全部歷史**
（back-adjust）——消除轉倉跳空、保持點數變化（Δ點）正確，使階梯/ATR/三關價與損益
計算不受換月污染。轉倉判定僅使用當日（收盤後）已知資訊，影響**次日起**的近月選擇
（看前偏誤紀律）。

**Why this priority**: 拼接規則是資料正確性的核心——錯的拼接會系統性污染所有回測
訊號與損益；它也是兩個資料源共用的邏輯（交叉驗證的前提）。

**Independent Test**: 拼接引擎單元測試（離線）——構造已知量交叉情境，驗證轉倉點、
差額平移數字、單向性與時序紀律，全部錨定例手算。

**Acceptance Scenarios**:

1. **Given** 近月與次月的日成交量序列在第 k 日交叉（次月 > 近月），**When** 產生連續序列，**Then** 第 k+1 日起採用次月價格，第 k 日（含）以前歷史以兩契約第 k 日收盤差回溯平移。
2. **Given** 交叉後次月量又短暫回落，**When** 判定近月，**Then** 不回切（單向規則）。
3. **Given** 連續序列任意相鄰兩日，**When** 計算點數變化，**Then** 不含轉倉跳空成分（Δ點 = 同一契約之真實變動或已調整之跨月變動）。
4. **Given** 截斷第 k 日之後的原始資料重新拼接，**When** 比對前 k 日連續序列，**Then** 近月選擇完全相同（時序紀律：判定不用未來資料；註：back-adjust 之平移基準隨資料尾端而異，比對以「近月選擇序列」為準）。

### User Story 3 - 雙源交叉驗證（Priority: P3）

系統可對重疊區間逐日比對主源（TAIFEX）與驗證源（FinMind）的**原始契約列**
（開高低收/結算價/成交量）：容差內視為一致；超差列產出告警報表供人工檢視。
驗證**不阻塞**入庫（主源為準、驗證源為哨兵）；驗證源不可用（無憑證/斷線/額度）時，
匯入照常執行、驗證跳過並如實記錄。

**Why this priority**: 使用者選定的資料品質保障（Q1=C）；獨立於主管線、可事後補跑。

**Independent Test**: 驗證器單元測試（離線 fixture）——一致樣本通過、注入差異產生
告警列、驗證源缺席時跳過且記錄。

**Acceptance Scenarios**:

1. **Given** 兩源同區間原始資料一致（容差內），**When** 執行交叉驗證，**Then** 報告全數通過、無告警。
2. **Given** 某日某契約收盤價被注入超差偏移，**When** 執行交叉驗證，**Then** 該列出現在告警報表（含兩源數值與差異）。
3. **Given** 驗證源憑證未設定，**When** 執行匯入（含驗證選項），**Then** 匯入正常完成、驗證步驟跳過並明確記錄「驗證未執行」。

### Edge Cases

- **back-adjust 後早期絕對價可能為負**：預期且無害——系統訊號（階梯/ATR/三關價/
  MSS/BOS）與損益全部只消費相對變化；spec 明示此性質，資料品質檢查不得誤殺負價
  （連續表之正價檢查對 back-adjust 序列調整為「非 NaN 且有限」）。
- **一般 vs 盤後時段**：2017-05 起每契約月份有兩列；日線一律取一般時段列。
- **來源格式改版**：欄位缺失/改名 → 解析 fail-fast 明確報錯，不得靜默容錯。
- **部分月份請求失敗**：有限次重試後中止回填並報告已完成區間；不留下靜默空洞。
- **結算週量交叉邊界**：量最大月規則在結算週自然轉倉（次月量放大）；無需另設結算日
  特例——規則單一、資料內生。
- **重複執行回填（冪等）**：同區間重跑 → 原始表以（日期×契約）為鍵覆蓋，不產生重複列。
- **驗證源額度耗盡**：視同不可用——跳過並記錄，不重試轟炸。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 提供 TAIFEX 官方主源匯入：歷史回填（指定區間、逐月分段、每請求間保守節流、Big5 解碼、過濾一般交易時段）與每日增量（官方公開當日端點）；請求失敗有限次重試後 fail-fast。
- **FR-002**: 原始「按契約月份」資料 MUST 分層儲存於獨立原始表（含開高低收、成交量、結算價、未平倉、契約月份）；表名經集中命名機制（`db_security`）核可（命名規則擴充以容納原始表）；同（日期×契約）重跑冪等覆蓋。
- **FR-003**: 連續月拼接 MUST 採成交量最大月規則：次月日成交量 > 現行近月 → 次日起轉倉至次月；單向不回切；每日判定；轉倉日以新舊契約當日收盤差**回溯平移全部歷史**（back-adjust）。
- **FR-004**: 拼接時序 MUST 遵守看前紀律（憲章 I）：第 k 日之近月選擇僅使用第 k−1 日（含）以前之量能資訊所做之判定結果（判定於收盤後、生效於次日）；截斷未來資料不得改變既往近月選擇序列。
- **FR-005**: 連續序列 MUST 存入既有期貨日線表（消費端——回測、監控、UI——零程式修改）；每次新轉倉觸發全歷史重算與整表覆蓋；連續序列 MUST 通過資料品質契約（時間嚴格遞增、無缺值、欄位完整；正價檢查對 back-adjust 序列放寬為有限數值）。
- **FR-006**: 系統 MUST 提供 FinMind 驗證源（同資料介面抽象；憑證走環境變數【安全鐵律】，不入 config 檔）。
- **FR-007**: 交叉驗證 MUST 為獨立步驟：對重疊區間逐日比對兩源原始契約列（開高低收/結算價/成交量），容差（config）內通過、超差列產出告警報表；驗證不阻塞入庫；驗證源不可用時跳過並如實記錄。
- **FR-008**: 組態 MUST 集中（憲章 V）：TXF instrument 之 source 切換為真源；容差、節流間隔、重試次數、回填起始（`backfill_start` **預設 1998-07-21 全歷史**，clarify 定案）等參數走 config + Pydantic 驗證；MTX 暫留 mock（真源接入僅需改 instrument 宣告，屬後續加法）。
- **FR-009**: 可重現性 MUST 保障（憲章 VI）：回填入庫後回測/監控完全離線；解析與拼接之單元測試用固定離線樣本（真實資料截取之 fixture）；需網路之端到端測試以專用標記隔離、預設跳過（CI 不對外部服務發請求）。
- **FR-010**: mock 與 csv adapter MUST 保留（測試/開發用途）；既有 mock 路徑測試行為不變。
- **FR-011**: 每條 Success Criterion MUST 對應至少一個自動化測試（憲章 III；network 標記之測試除 CI 外可手動執行驗收）。

### Key Entities

- **原始契約列**: 單一（交易日 × 契約月份）之市場資料——開高低收、成交量、結算價、未平倉、交易時段；儲存於原始表；交叉驗證與連續序列重建之依據。
- **連續序列**: 由原始列經近月選擇 + back-adjust 產生之單一 OHLCV 日線流；消費端唯一接觸的形態；絕對水位經平移（可能負值）、相對變化保真。
- **轉倉事件**: （日期、舊契約、新契約、收盤差額）四元組；近月選擇序列的變化點；back-adjust 的輸入。
- **交叉驗證報告**: 逐日逐契約之兩源比對結果——通過/超差、兩源數值、差額；超差列彙總為告警。
- **資料源組態**: TXF→真源宣告、容差/節流/重試/回填起始參數；驗證源憑證（環境變數）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001（真實資料端到端，network）**: 指定 3 個月區間回填 → 原始表與連續表筆數相符非零、連續序列通過品質契約、回測引擎直接跑通產生績效；每日增量將當日列附加入庫。
- **SC-002（拼接正確性）**: 離線錨定例——量交叉次日轉倉、差額平移數字逐位正確、單向不回切、截斷不變性（近月選擇序列）。
- **SC-003（解析正確性）**: TAIFEX 真實 Big5 樣本與 FinMind 樣本之離線 fixture → 欄位/型別/時段過濾全數正確；格式破壞 → fail-fast。
- **SC-004（交叉驗證）**: 一致樣本零告警；注入超差 → 告警列含兩源數值；驗證源缺席 → 跳過且記錄、匯入不受影響。
- **SC-005（零回歸）**: 全套既有測試綠；mock 路徑（含 003/008b 全部期貨測試）行為與數字不變。
- **SC-006（消費端最小改動，analyze H1 精修）**: 回測與 UI **零程式修改**消費真資料；監控之**期貨取數層**改為「讀取已入庫連續序列（歷史）＋官方當日端點（增量，每輪詢 1 請求）」——訊息格式、去重、mock 標示邏輯零改動；TXF 為真源時監控訊息無 mock 標示（離線注入驗證）。監控**不得**於輪詢中觸發歷史回填。
- **SC-007（品質契約）**: 連續序列通過 `validate_data_contract` 期貨門檻；無 NaN、時間嚴格遞增。
- **SC-008（冪等與韌性）**: 同區間重跑無重複列；請求失敗路徑明確報錯不靜默。

## Assumptions

- **端點事實（2026-07-17 實測）**: TAIFEX 批次下載 `POST /cht/3/futDataDown`（單次一個月、Big5、無驗證碼、需 UA）；OpenAPI `/v1/DailyMarketReportFut` 僅當日；歷史可溯 1998-07-21。TAIFEX 未公告 rate limit → 保守節流（預設 2 秒/請求，config 可調），10 年回填約 120 請求、一次性。
- **back-adjust 負價無害**: 系統全鏈（階梯、ATR、三關價、MSS/BOS、方向因子損益、保證金以點數×乘數計）僅消費相對變化或當期點位乘數；早期歷史絕對水位平移不影響任何訊號或績效語意。
- **量最大月規則之簡化**: 不設結算日特例——量交叉在結算週自然發生；此為確定性換月的刻意選擇（vs 固定結算日規則），與使用者 Q2=C 決策一致。
- **FinMind 為同源鏡像**: 交叉驗證主要捕捉「任一側的解析/傳輸錯誤」而非獨立行情源差異；此定位如實記載（哨兵，非仲裁者）。
- **範圍外**: 盤中資料（無免費歷史）、MTX 真源（加 instrument 宣告即可，後續）、券商 API、^TWII（僅可另作 sanity check，不進價格鏈）、組合回測期貨接入（護欄不動）。
- **002/007 FVG 懸案**: 真日線資料使 SC-003（FVG→P&L）可在更長歷史上重驗，但該重驗屬後續研究、不在本 spec 驗收。
