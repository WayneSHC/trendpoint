# 派工 Prompt 模板（Delegation Templates）

複製對應模板、填空、派工。`{...}` 為填空處。規則出處：`model-dispatch.md`。
每份模板都已內建：目標與動機、驗收條件、回報格式、路徑引號提醒、禁臆造條款。

**通用頁首（每個模板開頭都要貼）**：

```
工作目錄（絕對路徑，含中文與空格，Bash 中一律用雙引號包住）：
"{repo 絕對路徑}"
不確定的事標註「未確認」；禁止臆造檔案路徑、行號、API 行為。
引用任何 檔案:行號 前必須實際讀過該段。
```

---

## 1. 搜尋 / 定位（建議 model: sonnet；範圍明確且模式單純時 haiku）

```
【目標與動機】在 repo 中找出 {要找什麼，例如：所有讀取 config.yaml 的位置}。
動機：{為什麼找，例如：要新增參數，需要知道所有載入點以免漏改}。

【範圍】從 {起點目錄/檔案} 開始；{排除項，例如：排除 tests/ 與 data/}。
注意別名與間接引用：{已知別名，例如：AppConfig、load_config、get_settings}。

【驗收條件】列出每一個符合處的 檔案:行號 + 一行說明該處在做什麼。
自我檢查：用至少兩種不同的搜尋模式交叉驗證（例如符號名與字串字面值），
兩種結果的差集要逐一解釋。漏報視為失敗，誤報可接受。

【回報格式】直接回清單（檔案:行號 + 一行說明），開頭一句話回答總數。
不要貼大段程式碼。若結果 >30 筆，寫入 {落檔路徑} 並只回路徑 + 統計。
```

## 2. 實作（建議 model: sonnet；跨模組設計先派 Plan agent）

```
【目標與動機】實作：{功能一句話}。動機：{使用者要解決的問題}。
規格：{spec 路徑或條列需求}。

【約束】遵守 repo 憲法 ".specify/memory/constitution.md"：
訊號計算防看前偏誤（rolling 需 .shift(1)）、參數進 config/config.yaml + Pydantic、
禁止在熱路徑用純 Python 迴圈、UI 不內嵌演算法。
風格跟隨周邊既有程式碼。不要順手重構無關程式碼。

【驗收條件】
1. {可驗證的行為條件，例如：monitor_signals.py --once 在無憑證時輸出 Mock 訊息}
2. pytest -q 全綠；新行為有對應新測試 {測試檔路徑}。
3. {若動到訊號邏輯}：附前後回測對照（同資料、同成本），差異需可解釋。

【回報格式】開頭一句話：完成/未完成＋證據（測試輸出末行）。
然後：改了哪些檔（檔案:行號範圍 + 各一行動機）、做了哪些取捨、
哪些驗收條件未達成及原因。不要貼整份 diff。
```

## 3. 重構 / 批次修改（建議 model: sonnet；同型量產部分 haiku；加 isolation: worktree）

```
【目標與動機】重構：{改什麼，例如：把散落的路徑組裝改為共用 helper}。
動機：{例如：三處重複導致上次改漏}。**行為不得改變**。

【範圍】{明確檔案清單或 glob}。範圍外的檔案一律不碰，發現範圍外問題只記錄不修。

【驗收條件】
1. pytest -q 全綠，且不新增/修改/刪除任何測試（行為不變的證據）。
2. 全量驗證：用 grep 證明舊模式已清零：{舊模式的 grep 指令}。
3. 抽樣 read-back ≥3 個修改後的檔案，確認語意未變。

【回報格式】開頭一句話：完成/未完成。清單：每檔一行（檔案 + 改動類型）。
grep 清零證據與測試輸出末行。範圍外發現另列一節「未處理的觀察」。
```

## 4. 研究 / 查證（建議 model: sonnet；需要裁決多來源矛盾時 opus）

```
【目標與動機】查證：{問題，例如：LINE Messaging API push message 的免費額度與 429 行為}。
動機：{這個答案會決定什麼，例如：決定告警去重的節流參數}。

【方法】優先官方文件與一手來源；每個關鍵結論給來源 URL 與（若可能）發布/更新日期。
來源互相矛盾時並列雙方說法，不要自行擇一蓋掉。
區分三種狀態：已查證 / 多來源矛盾 / 查無資料——查無資料就直說，禁止用訓練記憶充當查證結果。

【驗收條件】{列出必答子問題}。每個子問題都有：答案 + 來源 + 信心（高/中/低）。

【回報格式】完整筆記寫入 {落檔路徑}；回覆只給：每個子問題一行結論＋信心，
與「會改變決策的發現」一節（若無則寫無）。
```

## 5. 審查（建議 model: opus 或當期最強；至少 sonnet + 第二意見）

```
【目標與動機】以 fresh context 審查 {對象：diff / PR / 檔案清單}。
動機：{例如：此變更動到部位計算，錯誤直接影響資金}。
你沒有參與產出，這是刻意的——不要向做工方求證，只看證據。

【審查維度】依序：(1) 正確性（邊界、None/NaN、時序對齊、look-ahead）；
(2) 安全（憑證、SQL 注入、路徑注入）；(3) 與規格/憲法的一致性
（".specify/memory/constitution.md"）；(4) 測試是否真的驗到宣稱的行為
（讀測試本體，不是看測試名稱）。

【驗收條件】每條 finding：檔案:行號 + 嚴重度（Critical/High/Medium/Low）+
失敗情境（什麼輸入/狀態會炸）+ 建議修法。
每個維度都要有結論，沒發現就明寫「未發現」。
對「看起來可疑但未確認」的項目：標 PLAUSIBLE，並寫出驗證方法。

【回報格式】severity 統計一行 + finding 清單（嚴重度排序）。
>10 條時完整版落檔 {路徑}，回覆只留 Critical/High。
最後一句話下總判斷：可合併 / 修完再合 / 不可合併。
```

---

**模板維護**：踩到「模板沒涵蓋導致派工失敗」的坑，按 `maintenance-protocol.md`
把缺的條款補進對應模板，不要另起爐灶開新模板（五種型態夠用，先問自己是不是填空沒填好）。
