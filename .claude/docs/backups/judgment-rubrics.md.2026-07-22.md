# 判斷力外化（Judgment Rubrics）

把「憑感覺」的判斷寫成可執行的判準。每條附正例（✅ 該這樣判）與反例（❌ 不該這樣判）。
拿不定主意時：先查本檔有沒有對應判準；有就照做，沒有就套第 6 節的誠實條款。

## 1. 何時該升級模型（或加派第二意見）

**判準**：滿足任一即升級——
(a) 同一子任務兩次獨立嘗試都失敗或互相矛盾；
(b) 錯誤後果不可逆或代價高（會動到訊號邏輯、資金計算、對外發布、刪除）；
(c) 任務需要跨 >3 個模組的整體設計，而不是局部修改；
(d) 你發現自己在「猜」根因而不是「推」根因。

- ✅ 正例：walk_forward 回測結果與單標的回測不一致，sonnet agent 查了兩輪，
  一次說是滑價設定、一次說是資料對齊，結論互斥 → 升級 opus，附上兩份失敗報告。
- ❌ 反例：改 README 錯字改了兩次才對 → 這是粗心不是能力上限，不升級，
  改成「改完立刻 read-back」即可。升級是為能力不足，不是為流程草率。

## 2. 何時算真的完成

**判準**：同時滿足才算完成——
(a) 原始要求逐條對照過（不是憑印象「都做了」）；
(b) 有可觀察證據：測試綠、實跑輸出、read-back 結果，寫進回報；
(c) 產出經 fresh-context 驗收（見 model-dispatch.md 第 6 節）——自己的檢查不算數；
    免驗例外：≤3 行、且不涉及訊號邏輯/資金計算/授權/安全的修改，可自行 read-back
    代替（與 maintenance-protocol.md 第 5 節同準）；
(d) 交易邏輯變更額外要求：`tests/test_lookahead_bias.py` 涵蓋新訊號＋前後回測對照已附。

- ✅ 正例：「加上 Telegram 推播降級邏輯」→ 改完後實跑
  `python monitor_signals.py --test-alert`（無憑證環境）確認走到 Mock 分支，
  貼出該行輸出，才回報完成。
- ❌ 反例：「pytest 全綠所以完成」——但原始要求還有「更新 README 的設定說明」，
  沒對照清單就宣告完成。**測試綠 ≠ 需求滿足**。

## 3. 何時該停下來問使用者

**判準**：滿足任一就停——
(a) 不可逆且超出原始委託範圍：刪除非自己建立的檔案、force push、對外發布/推播真實訊息、
    改動 `~/.claude/settings.json` 這類使用者層設定；
(b) 兩個合理方案的取捨取決於使用者沒說出口的偏好（例如授權條款的商業意圖、
    要不要把 repo 搬出 Google Drive）；
(c) 發現使用者的前提可能錯了（要求修 A，但證據顯示問題在 B）——先回報發現，不要默默改 B。

**反向判準**（不要停）：可逆、在委託範圍內、有慣例預設值的決定，選定、註明、繼續做。
問題攢成一批問，不要一題一停。

- ✅ 正例：使用者說「把過時檔案清掉」，你認定 `TrendPoint_OpenSpec.md` 過時，
  但它是歷史規格且非你建立 → 停，列出清單請使用者勾選。
- ❌ 反例：「幫我加個測試」→ 停下來問「測試函式要叫什麼名字？」——命名有慣例，
  自己決定。把使用者當成只處理例外的主管，不是逐步核可的監工。

## 4. 什麼訊號代表方向錯了（該換路，而非重試）

**判準**：出現任一訊號，停止在原路上加力——
(a) 連續兩次修復都是在修「上一次修復引入的問題」（修補鏈）；
(b) 為了讓方案成立，需要的特例/旁路越來越多（第 3 個 workaround 就是訊號）；
(c) 每輪修改後失敗的測試「換了一批」而不是「變少」；
(d) 你開始想放寬驗收條件來讓現狀通過。

**換路動作**：回到最後一個已知正確狀態（git stash / checkout），寫下 3 行
「這條路為什麼死」，換一個假設重來或開 fresh-context agent 重來。
沉沒成本不是理由：已花的 token 不會因為繼續錯下去而回收。

- ✅ 正例：為了讓 numba 降級測試過，先 mock 了 import、再 patch 環境變數、
  第三次要改 conftest → 停。真正問題可能是降級偵測邏輯本身，換路去讀
  `ladder_system.py` 的 jit fallback 實作。
- ❌ 反例：測試失敗一次，訊息明確指出是參數名打錯 → 這不是換路訊號，改掉重跑即可。
  換路判準針對「重複失敗的模式」，不是「單次失敗」。

## 5. 品質底線怎麼驗

各類產出的最低驗法（低於此不准交付）：

| 產出 | 底線驗法 |
| :--- | :--- |
| 程式碼 | `pytest -q` 全綠 + 實跑受影響的入口腳本一次；綠燈含 skip 時用 `-rs` 逐條看跳過理由——該跑卻跳過等於沒驗到（見下方踩坑教訓） |
| 訊號/回測邏輯 | 上行 + 前後回測對照（同資料同成本）+ look-ahead 測試涵蓋 |
| 文件/制度檔 | fresh-context agent read-back：只給文件不給脈絡，要求它複述規則並舉一個執行例；複述錯 = 文件寫得不夠明確 |
| 對外文案/授權/安全 | 第二意見（獨立 agent 重做判斷）結論一致才放行 |
| 批次修改 | 抽樣 read-back（≥3 個樣本 + 全量 grep 驗證模式） |

- ✅ 正例：批次把 30 個檔案加上授權標頭 → grep 統計標頭出現次數 = 檔案數，
  再抽 3 檔 read-back 確認位置與格式。
- ❌ 反例：「diff 看起來每個檔都改到了」→ 視覺掃過不是驗證。

**踩坑教訓**：

- 坑：同一份程式碼，乾淨工作區 `pytest -q` 全綠，跑過 `python monitor_signals.py --once`
  之後卻變 1 failed（`test_portfolio_backtest_execution`）。
  為什麼：該測試的前提檢查（guard）只驗「`trendpoint.db` 檔案存在」，而 monitor 初始化
  去重表時的 `sqlite3.connect` 會順手建出只含 `sent_alerts` 的空 db——guard 的判準弱於
  被測程式的真實前提（至少一張非空 `stock_*_daily`），於是該跳過的情境變成失敗。
  下次怎麼做：guard 的判準一律等同被測程式的前提，禁用「檔案存在」這類代理指標；
  需要真實 DB 的測試直接用 `conftest.py` 的 `tickers_with_data` fixture：
  `tickers_with_data(db_path, tickers)` 回傳其中「有非空日線表」的標的，空清單就
  `pytest.skip`（涵蓋測試見 `tests/test_db_preconditions.py`）。改動 guard 後跑三態
  驗證——無檔／有檔但無資料／有資料——第三態必須確認測試**實際執行**
  （`pytest -rs` 且該測試不在 skip 清單），否則「永遠跳過」的 guard 一樣是綠的。

## 6. 誠實條款：這套制度補不了的東西

拆解、驗證、多樣本評審能補**執行品質**；補不了**模糊題與品味判斷**
（策略參數的合理範圍、文件的輕重取捨、「這個抽象對不對」）。遇到時按序處理：

1. 把模糊題翻成可驗證題（「這參數合理嗎」→「跑消融測試量化貢獻」）。翻得出來就做。
2. 翻不出來 → 升級到當期最強模型，或多答案評審（3 個獨立方案 + 1 個評審）。
3. 還是拿不準 → 明說「這題是品味判斷，我的建議是 X，信心中等，理由如下」，
   讓使用者裁決。**禁止**把低信心結論包裝成高信心交付。
4. 查不到的事實標註「未查證」，不要編造（版本號、API 行為、法律條款尤其如此）。
