# 模型調度守則（Model Dispatch）

給主對話模型（指揮官）的派工規則。目的：保住主對話的 context 與判斷力，
把消耗 token 的執行工作推給 subagent。病灶依據見 `harness-diagnosis.md` 第 2 名。

## 1. 指揮官不下場

以下工作**一律派 subagent**，主對話不自己做：

| 工作型態 | 派給誰 | 理由 |
| :--- | :--- | :--- |
| 掃 repo、找符號、跨檔搜尋 | `Explore` agent | 搜尋過程的中間結果不該進主對話 |
| 整檔閱讀 >2 個檔、或任何 >20KB 的檔 | `Explore` 或 general-purpose | 同上 |
| 查網頁 / 查文件 | general-purpose（帶 WebFetch/WebSearch） | 網頁原文極長 |
| 批次改檔（>3 檔的機械性修改） | general-purpose，必要時 `isolation: worktree` | 改壞可丟棄 |
| 跑長測試 / 長回測並解讀輸出 | general-purpose | 原始輸出長 |
| 驗收別人的產出 | **fresh-context** agent（見第 6 節） | 不自驗 |

主對話**可以**自己做：≤2 個中小檔的精準讀取（先 Grep 定位）、單檔小修改、
git 操作、跑 `pytest -q` 這類輸出短的指令、以及所有「決策」。

**違規訊號**：主對話連續三次工具呼叫都是整檔 Read 或探索式 ls/grep → 停下，改派工。

## 2. 派工三件套

每個 subagent prompt 必含三件事，缺一件就是壞派工：

1. **目標與動機**：要什麼＋為什麼要（動機讓 agent 在邊界情況做對取捨）。
2. **驗收條件**：可檢查的完成定義。「找出所有呼叫點」→「列出每個呼叫點的檔案:行號，
   並說明各自傳入的參數型別；漏一個算失敗」。
3. **回報格式**：明確規定回什麼、不回什麼（見第 4 節回報合約）。

另加兩個常備條款：工作目錄的絕對路徑＋「路徑一律雙引號」提醒；
以及「不確定就標註『未確認』，禁止臆造檔案路徑與行號」。

## 3. 顯式指定 model 與 effort

Agent 工具有 `model` 參數（`haiku` / `sonnet` / `opus`）；不指定則用該 agent
定義自帶的模型，無定義才繼承主對話模型。
Workflow 的 `agent()` 另有 `effort` 參數（low→max）。**每次派工都要有意識地選，
不要用預設值矇混**：

| 任務 | model | 依據 |
| :--- | :--- | :--- |
| 機械性批次工作：改名、格式化、read-back 驗證、簡單抽取 | `haiku` | 便宜快，錯了易發現 |
| 標準工作：搜尋、有明確 spec 的實作、照 checklist 審查 | `sonnet` | 預設工作馬 |
| 高難度：模糊除錯、演算法設計、高風險變更的對抗審查 | `opus`（或當期最強可用模型） | 錯誤成本高 |

effort（可用時）：機械工作 `low`；預設 `medium`~`high`；對抗驗證與交易邏輯正確性審查
用最高檔。若 harness 不支援 effort 參數，就在 prompt 裡文字要求（「逐行檢查、
每個結論附證據」），效果次之但可用。

## 4. 回報合約

Subagent 的最終回覆只准回：**結論 + 檔案:行號 + （若有長產物）落檔路徑**。

- 長產物（報告、大 diff 說明、掃描結果）一律寫成檔案，回覆只傳路徑＋≤10 行摘要。
- 禁止在回覆貼整份檔案內容、整段程式碼（≤10 行的關鍵片段除外）。
- 回覆開頭第一句必須直接回答派工的問題（找到/沒找到、通過/不通過）。
- 派工時把這段合約原文貼進 prompt，不要假設 agent 自己知道。

## 5. 升降級路徑

- **升級**：sonnet agent 兩次嘗試同一子任務都失敗、或回報內容自相矛盾 →
  改派 opus 重做**並附上失敗紀錄**（讓它知道別人試過什麼）。何時該升級的完整判準
  見 `judgment-rubrics.md` 第 1 節。
- **降級**：任務被證明是機械性的（第一個樣本做對後其餘同型）→ 剩餘量派 haiku。
- **放棄重派**：同一 agent 在一個任務上來回修 >3 輪 → 丟棄該線，
  開 fresh-context agent 重來，prompt 附上「已知死路清單」。續聊用 SendMessage
  只適合追問澄清，不適合救一個已經歪掉的 agent。

## 6. 驗證不自驗

做的人不驗收自己的產出。驗收一律派 **fresh-context** agent——注意：fresh-context
不是 agent 類型名稱，指「新開一個 Explore 或 general-purpose agent」：不沿用先前
對話、不附產出者的推理，只給驗收條件。按產物類型：

- **檔案/文件**：read-back——驗收者實際打開檔案，逐條對照驗收條件，回報每條通過/不通過。
- **程式碼**：跑測試或實跑（`pytest -q`、實際執行入口腳本），不接受「看起來對」。
  影響訊號邏輯的變更，驗收者要跑前後回測對照。
- **高風險判斷**（交易邏輯、授權、安全、不可逆操作）：第二意見——再派一個 agent
  獨立重做判斷，兩者結論不一致時交回主對話裁決或問使用者；或多答案評審：
  派 2~3 個 agent 各自給方案，再派一個評審 agent 選優並說明淘汰理由。

驗收不過 → 退回原做工線修（附驗收報告），修完再驗，最多兩輪；兩輪不過走第 5 節放棄重派。
